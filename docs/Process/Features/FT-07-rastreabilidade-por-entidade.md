# [FEATURE] FT-07 — Entity and Correlation Modification Traceability

## Description

**Wave 2 | Lean Inception: Traceability**

This feature enriches the API and the web interface with traceability capability in two complementary dimensions:

**1. Entity traceability** (based on the `index` field): given a specific record for an entity (identified by the `index` field of the AuditEvent), displays the complete history of operations on that object — who changed it, what was changed, when, and in which version. Allows De Marco and Letícia to reconstruct the timeline of a specific piece of data within a single source system.

**2. Correlation traceability** (based on `correlationId`): given a distributed operation identifier, displays all events recorded in different microservices that are part of the same business transaction. Allows identification of the impact of a single user action that propagates through CRM, Finance, Inventory, and other services, unifying scattered evidence into a single cross-service timeline.

The `index` field of the AuditEvent contract was designed for entity traceability: the client system determines which fields uniquely identify the audited record (e.g.: `{"orderId": "9872"}`) — RavenLedger uses these fields to correlate operations on the same object over time. The `correlationId` (a `metadata` field) is generated by the client system to propagate the context of a distributed operation between services.

**Business scenarios:**
- **Entity history:** De Marco provides entity "orders" and index `orderId=9872` → chronological list of all operations on that order, with diff between versions.
- **Distributed traceability:** De Marco provides `correlationId` "op-7263" → all events from CRM, Finance, and Inventory generated in the same transaction, ordered by `generated_at`.
- **Entity with no history:** Index not found → `200 OK` with empty list.
- **Multiple index fields:** `customerId=123` + `productId=456` → correlation by field combination.
- **Diff between versions:** Comparison between two consecutive records of the same entity → changed fields highlighted.

**Success indicators:** Reduce the time to reconstruct an entity's history from hours (manual) to seconds (automated search). Enable cross-service traceability without the investigator needing to cross-reference logs from different systems.

## Technical Description

**Affected services:** `raven-ledger.api` (new endpoints), `raven-ledger.register` (indexing), web interface (new screens)

**Contract:** `docs/systemDesign/audit-api.openapi.yaml` (updated — version 1.1.0)

**New endpoints in `raven-ledger.api`:**

```
GET /api/v1/audit-entries/history
  Query params:
    entity       (string, required)
    domain       (string, optional)
    index_field  (string, required) — name of the field in the index object
    index_value  (string, required) — field value
  Response: list ordered by generated_at (asc), with diff field between consecutive records

GET /api/v1/audit-entries/{id}/diff/{other_id}
  Compares two specific records
  Response: { added: {...}, removed: {...}, modified: { field: { from: ..., to: ... } } }

GET /api/v1/audit-entries/trace
  Query params:
    correlation_id (string, required) — distributed operation identifier
    page           (int, default 1)
    size           (int, default 100, max 500)
  Response:
    {
      "correlationId": "op-7263",
      "entriesCount": 5,
      "sources": ["https://crm.example.com", "https://financial.example.com"],
      "items": [...]  -- same fields as GET /api/v1/audit-entries
    }
  400 if correlation_id is absent
```

**Indexing in LedgerDatabase** (migration to be created by `raven-ledger.register` as part of this feature):

```sql
-- GIN index for searching by index_payload fields
CREATE INDEX idx_ledger_entries__index_payload ON ledger_entries
  USING GIN (index_payload)
  WHERE index_payload IS NOT NULL;

-- Composite index for entity traceability
CREATE INDEX idx_ledger_entries__entity_domain ON ledger_entries (entity, domain, generated_at);
```

> The index for `correlation_id` was already created in FT-03 (`idx_ledger_entries__correlation_id`). The `/trace` endpoint uses it directly.

**Diff logic:** Field-by-field comparison between `data_payload` (JSONB) of two consecutive records of the same entity key. The diff is calculated in the API (not stored), as it depends on which pair of records the user is comparing.

**New screens in the web interface:**

- "Entity History" screen (`/audit-entries/history`):
  - Form: `entity`, `domain` (optional), `index_field`, `index_value`
  - Vertical timeline with events in chronological order
  - Each item: `operationType`, `userName`, `generatedAt`
  - Click on a timeline item: side panel with diff between this and the previous record
  - Modified fields highlighted (green = added, red = removed, yellow = modified)

- "Correlation Traceability" screen (`/audit-entries/trace`):
  - Field: `correlationId` (required)
  - Timeline grouped by source service (`source`/`domain`)
  - Each entry: service, entity, operation type, user, `generatedAt`
  - Allows the expert to see the distributed impact of a single business transaction

**Structured logs:**
- `entity_history_query` → `entity`, `index_field`, `result_count`, `latency_ms`
- `entry_diff_computed` → `entry_id_a`, `entry_id_b`, `changed_fields_count`, `latency_ms`
- `correlation_trace_query` → `correlation_id`, `result_count`, `sources_count`, `latency_ms`

## Acceptance Criteria

- [ ] `GET /api/v1/audit-entries/history?entity=X&index_field=Y&index_value=Z` returns list ordered by `generated_at`
- [ ] Returned records belong only to the provided entity + index field + value combination
- [ ] Entity with no history returns empty list (no error)
- [ ] `GET /api/v1/audit-entries/{id}/diff/{other_id}` returns structured diff between two records
- [ ] GIN index created and used in the history query (no full table scan)
- [ ] Response < 200ms for entities with up to 1,000 records
- [ ] Web interface displays timeline with diff between consecutive versions
- [ ] Modified fields are visually highlighted in the interface
- [ ] `entity` and `index_field` are required → `400 Bad Request` if absent
- [ ] `GET /api/v1/audit-entries/trace?correlation_id=X` returns all records with that `correlationId`, ordered by `generated_at`
- [ ] `/trace` response includes the list of distinct `sources` found
- [ ] `correlation_id` absent in `/trace` → `400 Bad Request`
- [ ] `/trace` response < 200ms for operations with up to 500 records
- [ ] Web interface displays the correlation traceability screen with timeline grouped by service

## Priority

2

## Risk

2 — Medium. Searching a JSONB field (`index_payload`) can be costly without an adequate GIN index. High volume (millions of records) can degrade performance if the index does not cover the access pattern. The `/trace` endpoint depends on the partial index `idx_ledger_entries__correlation_id` defined in FT-03 — any degradation in that index directly impacts cross-service traceability. Test with representative volume before enabling in production.
