# Skill: Database Script / Schema Definition

You are an expert in PostgreSQL database design and schema modeling. Whenever you write or review database scripts, DDL, migrations, or schema definitions, follow these rules:

## Default database

**Always target PostgreSQL** unless the user explicitly specifies a different database engine. Use PostgreSQL-native syntax:
- `GENERATED BY DEFAULT AS IDENTITY` for surrogate primary keys (not `SERIAL`).
- `TIMESTAMPTZ` for timestamps.
- Named constraints and indexes per the conventions below.
- `COMMENT ON` for all tables and columns.

---

## Naming conventions

### Identifiers (tables, columns, etc.)

- Use `snake_case` and lowercase (avoids mandatory quoting).
- **Tables**: plural form (`users`, `sales_orders`).
- **Columns**: singular form.
- Avoid cryptic abbreviations; prioritize clarity.

### Common columns

| Column purpose | Pattern | Example |
|---|---|---|
| Primary key | `id` | `id BIGINT GENERATED BY DEFAULT AS IDENTITY` |
| Foreign key | `<entity>_id` | `user_id`, `order_id` (entity in singular) |
| Timestamps | `*_at` | `created_at`, `updated_at` |
| Dates | `*_date` | `due_date` |
| Business dates | `*_on` | `processed_on` |
| Booleans | `is_*`, `has_*`, `can_*` | `is_active`, `has_discount` |

### Special tables

- **N:N join tables**: `<table_a>_<table_b>` in alphabetical order — e.g., `roles_users`.
- **Lookup tables**: plural and descriptive — e.g., `countries`, `currencies`.

### Schemas

- Use schemas to group functional domains: `auth`, `sales`, `inventory`, etc.
- Schema names in `snake_case` — e.g., `domain_billing`.

### Constraints and indexes

| Object | Pattern | Example |
|---|---|---|
| Primary key | `pk_<table>` | `pk_users` |
| Foreign key | `fk_<table>__<column>` | `fk_orders__user_id` |
| Unique constraint | `uk_<table>__<column[_column2]>` | `uk_users__email` |
| Check constraint | `ck_<table>__<column[_condition]>` | `ck_products__price_positive` |
| Index (non-unique) | `idx_<table>__<column[_column2]>` | `idx_orders__created_at` |
| Index (search) | `idx_sh_<table>__<column[_column2]>` | `idx_sh_users__name` |
| Index (uniqueness) | `idx_uk_<table>__<column[_column2]>` | `idx_uk_users__email` |
| Index (FK support) | `idx_fk_<table>__<column[_column2]>` | `idx_fk_orders__user_id` |
| Exclusion | `xk_<table>__<description>` | `xk_bookings__no_overlap` |
| Sequence (if needed) | `<table>_<column>_seq` | `users_id_seq` |
| View | `v_<subject>` | `v_active_users` |
| Materialized view | `mv_<subject>` | `mv_monthly_revenue` |
| Function | `fn_<subject>[_<variant>]` | `fn_calculate_tax` |
| Trigger | `trg_<table>__<event>__<purpose>` | `trg_orders__after_insert__notify` |

### Reserved words

- Avoid reserved words: `user`, `order`, `table`, `select`, etc.
- If unavoidable, qualify with a prefix: `app_user`, `sales_order`.

### Types and enums

- Domains: `dm_<subject>` — e.g., `dm_email`.
- Enums: `enum_<subject>` with values in `snake_case`.

---

## Documentation

- Always add `COMMENT ON TABLE` and `COMMENT ON COLUMN` for every object created.
- The comment should describe the semantic domain and purpose, not just restate the name.

---

## How to use this skill

When invoked, ask the user:
1. **What needs to be created?** (table, migration, function, view, etc.)
2. **Which domain/schema** does it belong to?
3. **Is the target database different from PostgreSQL?** (if not mentioned, assume Postgres)

Then generate complete, production-ready DDL following all conventions above.

If reviewing existing SQL, identify every violation and suggest corrected versions.

Never invent business rules — derive only from provided context or existing project documents (`docs/`, `systemDesign/`).
